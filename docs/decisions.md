# 需求取捨決策

> 這份文件可以由 Codex 先產生草稿，但最後必須由你刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：資訊整理者是資訊流的中樞。只有先建立一套可靠的「資料生命週期」流程（標記衝突、過期、隱私權處理），前端才有正確的資料餵給行動者，回報者才會收到有意義的提醒。

## 這個使用者最想完成什麼

- 降低誤判風險。將混亂、未核查的原始消息快速過濾，標記其可信度與時效性，加上人工審查備註，轉換為低風險的「待查證/已查實」草稿，提供前線安全出發的參考依據。

## 這個使用者最怕什麼錯誤

- AI腦補以及浪費人力。AI 智慧分析過度「腦補」，自動將模糊或無時間戳記的流言（M-004、M-005）標記為「高品質、可直接轉成任務」，導致志工前往錯誤的現場浪費寶貴人力。

## 我今天暫時不處理的需求

請至少寫一項。

- **回報端地圖自動定位與地址解析**：在回報端，暫時不處理模糊地址的自動地圖定位。此需求會增加回報者負擔，且在未經查核前自動定位會誤導志工；我們暫時維持文字描述，並在整理端進行人工查實後才提供確切位置標記。

## 我不採用的 AI 建議

請至少寫一項，並說明原因.

- **AI 建議 1**：AI 自動將通報分類並「自動補全」任務建議。
  - _不採用原因_：這導致了對「信心程度」與「類型標籤」的盲目依賴。未來應將 AI 定位為「輔助標記」而非「自動決策」，所有 AI 產生的建議都必須以「建議」形式呈現，並要求人工確認後才能轉為正式狀態。
- **AI 建議 2：依據 sourceType（資訊來源類型）將官方公告直接預設為 verified（已驗證）**。
  - _不採用原因_：官方封路消息也有極強的時效性（如過期截圖 M-005）。如果直接判定為 verified，會誤導前線志工。所有來源均需保留人工核實點。

## 必須保留人工確認的判斷

- 決定該筆資料是否可以轉為「已確認並可行動」的最終狀態，所有資訊流轉必須經過明確的「人工確認」環節。
- 對於代通報長者個資（M-011），人工必須確認「當事人是否同意公開完整資訊」，否則需手動遮蔽。
- **當多筆通報來源有內容衝突時（如側門是否淹水 M-006），禁止 AI 自動合併或覆蓋，必須保留人工介入判定並在 UI 同步警示**。

## 今天明確不做什麼

- 不寫後端資料庫架構：目前的重點應放在前端的「資訊流轉邏輯」與 UI 的「狀態暗示」（所有草稿狀態持久化儲存在 localStorage）。
- 不上真實地圖定位：為了避免隱私外洩風險與不準確的地標誤導，v1 僅處理文字資訊的邏輯與時序狀態（不引入真實外部地圖）。
- 不使用「自動化」發布：所有資訊流轉必須經過明確 of「人工確認」環節，拒絕將 AI 自動產生的標籤直接呈現為「高品質」或「已完成」。
- **不自行修改核心 `CommonRecord` 資料架構規格：確保與 Phase 0 的 messy-reports 測試資料保持相容，不破壞核心資料合約**。

## 我仍不確定的地方

- 對於完全沒有聯絡電話且地點模糊的通報（如 M-003 聽說老街需要水電），除了將其標記為「無法查核/不處理」外，是否需要有「順路探查」的次級狀態？這是否會引導志工前往不安全的地方？
- **如何在前端 localStorage 儲存電聯查核紀錄，且確保查核人與被查核者的聯絡個資在多個協作者介面之間不被無授權外洩？其隱私與稽核邊界該如何拿捏？**
