# Phase 0 Observations

## 我們做出了什麼？

- 災害資訊整理工作台前端原型，提供原始資訊列表、安全預設邊界顯示，以及一鍵建立、編輯、刪除和重設草稿的功能（Draft CRUD）。草稿中可標記可能類型、信心程度、下一步建議、阻礙點（Blockers）與人工審查備註，清楚標明安全邊界。

## 哪些資料讓 agent 很容易整理？

- 結構化良好、由現場值守志工回報且有明確數量與時效的資訊（例如 M-010 盤點雨鞋數量與飲用水狀態）。這類資訊來源可靠，可直接轉為地點更新建議。

## 哪些資料讓 agent 亂補或誤判？

- 缺乏時間戳記的社群截圖或流言（例如 M-004、M-005）。Agent 容易自行推測並建議將其直接列為已驗證任務或公告，但實際上面臨截圖日期不詳、社群消息與現場實況不對齊的重大風險。

## 哪些資訊不能直接變成任務？

- 地點極度模糊（M-001 「老雜貨店後面」）、具時效性但已過期（M-002 早上有雨鞋下午未知）、或是片面的第三方代通報且無親友位置（M-012 家屬來電且位置不明）。

## 哪些資訊的操作者不是當事人？

- 志工代不方便使用手機的長者轉述（M-011，涉及隱私與是否同意公開地址問題）、外地家屬來電代親友尋求協助（M-012，無法聯繫親友本人確認）。

## 我們剛剛偷偷假設了什麼？

- 假設了志工或家屬的轉述是 100% 正確的；假設了貼文中的人名與電話真實且可信；假設了系統後台總是有足夠的人工審核人員能隨時撥打電話確認資訊。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 建立明確的資料格式（Schema）約束（如經緯度坐標、電聯紀錄、隱私授權欄位），並實作流程狀態機（State Machine），定義原始資料、草稿、候選、已發佈任務的嚴格流轉規則，非單純由 UI 判定。

## 我們需要更清楚資料格式的地方

- 什麼是「確切位置」（如何利用圖資或地標定義，防範『老街附近』、『第二排住家』等模糊字眼）；「關係人角色」的格式定義（回報者、當事人、現場值守者之權限與隱私設定）。

## 我們需要人工確認的地方

- 涉及當事人個資與隱私公開意願時（M-011）；資訊出現嚴重衝突時（M-006 關於側門是否適合作為集合點的淹水爭議）；以及時效性高且會影響物資分配的傳言（M-004 群組呼籲直接拿雨鞋）。

## 隱私防護的安全極限與 AI 貼標籤風險反思 (V1 角色分流反思)

### 1. 「前端假遮蔽」的安全性與技術極限

- **安全漏洞**：在當前沒有後端資料庫的起步專案 (starter project) 階段，系統對居民電話、精確住址的「遮蔽模糊化」是在前端 React 渲染層實作的。雖然這能防範一般大眾路過窺探，但實際上所有的原始明文個資都已完整下載到前端瀏覽器的 JS 記憶體中。稍懂技術的使用者只要按 F12 開啟開發者工具即可看光個資。
- **改善方向**：在未來的正式系統中，去識別化與隱私遮蔽必須**強制移入後端 API 伺服器**。未經實名驗證的訪客，後端在回傳通報列表時就應直接抹除敏感資訊，保證原始明文資料絕不下載至客戶端。

### 2. 形同虛設的前端身分驗證

- **安全漏洞**：我們目前在整理端（`STAFF2026`）與行動端（`VOLUNTEER`）實作的口令驗證，其密碼是直接寫死在前端程式碼中。這容易因為金鑰外洩或代碼洩漏而形同虛設，且完全缺乏對志工真實身分的認證與事後日誌審計 (Access Log Audit) 機制。
- **改善方向**：未來必須引進實名 OAuth 認證與動態權限模型。每當有人點選「同意承諾書並解鎖個資」時，後台必須對存取行為進行持久化存檔，包括「誰、在什麼時間、存取了哪位居民的什麼欄位」，以便於發生洩密事件時追究安全法律責任。

### 3. AI 推估「腦補」對居民權益的貼標籤風險

- **安全風險**：AI 智慧分析存在嚴重的「腦補」與誤判偏誤。如果 AI 僅因通報語意模糊，就把一筆真實且緊急的居民求助信心度判定為「低 (Low)」，或者歸類為「流言/Do not use」，這筆通報在後台就可能被忽略，直接導致居民的生存救援權利受損。
- **改善方向**：系統必須堅持「安全預設與人工二核」原則。AI 的任何建議（信心度、區域推估）僅可作為「黃色虛線閃爍框」的 UI 提示，不進行自動儲存。且對於被標為「低信心度」的資料，依然要保留由整理者一鍵切換排查的備用入口，防範 AI 偏誤對受災個體權益的侵害。
