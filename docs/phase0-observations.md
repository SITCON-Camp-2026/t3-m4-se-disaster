# Phase 0 Observations

## 我們做出了什麼？

- 災害資訊整理工作台前端原型，提供原始資訊列表、安全預設邊界顯示，以及一鍵建立、編輯、刪除和重設草稿的功能（Draft CRUD）。草稿中可標記可能類型、信心程度、下一步建議、阻礙點（Blockers）與人工審查備註，清楚標明安全邊界。

## 哪些資料讓 agent 很容易整理？

- 結構化良好、由現場值守志工回報且有明確數量與時效的資訊（例如 M-010 盤點雨鞋數量與飲用水狀態）。這類資訊來源可靠，可直接轉為地點更新建議。

## 哪些資料讓 agent 亂補或誤判？

- 缺乏時間戳記的社群截圖或流言（例如 M-004、M-005）。Agent 容易自行推測並建議將其直接列為已驗證任務或公告，但實際上面臨截圖日期不詳、社群消息與現場實況不對齊的重大風險。

## 哪些資訊不能直接變成任務？

- 地點極度模糊（M-001 「老雜貨店後面」）、具時效性但已過期（M-002 早上有雨鞋下午未知）、或是片面的第三方代通報且無親友位置（M-012 家屬來電且位置不明）。

## 哪些資訊的操作者不是當事人？

- 志工代不方便使用手機的長者轉述（M-011，涉及隱私與是否同意公開地址問題）、外地家屬來電代親友尋求協助（M-012，無法聯繫親友本人確認）。

## 我們剛剛偷偷假設了什麼？

- 假設了志工或家屬的轉述是 100% 正確的；假設了貼文中的人名與電話真實且可信；假設了系統後台總是有足夠的人工審核人員能隨時撥打電話確認資訊。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 建立明確的資料格式（Schema）約束（如經緯度坐標、電聯紀錄、隱私授權欄位），並實作流程狀態機（State Machine），定義原始資料、草稿、候選、已發佈任務的嚴格流轉規則，非單純由 UI 判定。

## 我們需要更清楚資料格式的地方

- 什麼是「確切位置」（如何利用圖資或地標定義，防範『老街附近』、『第二排住家』等模糊字眼）；「關係人角色」的格式定義（回報者、當事人、現場值守者之權限與隱私設定）。

## 我們需要人工確認的地方

- 涉及當事人個資與隱私公開意願時（M-011）；資訊出現嚴重衝突時（M-006 關於側門是否適合作為集合點的淹水爭議）；以及時效性高且會影響物資分配的傳言（M-004 群組呼籲直接拿雨鞋）。
