# Main Agent 訪談彙整

> 這份文件可以由 Codex 先產生草稿，但你必須刪除、修改或補充。請保留人類確認痕跡。

## 三個 persona 的共同需求

- **對「資訊時效性」的極度關切**：不論是回報者、整理者還是前線行動者，都極度關心資訊的更新時間。若無時間標記，大家完全無法判斷物資是否已被拿光（M-002）或路障是否已清除。
- **對「資訊來源可信度」的透明呈現**：需要清晰分辨這筆通報是「親眼所見（第一手）」還是「聽人轉述（二手傳言）」，避免將社群貼文（M-004）或未查證的截圖當成事實。
- **對「地點描述精確度」的警示**：如果通報地點過於模糊（如 M-001 「老雜貨店後面」），回報者應被提醒，整理者需有管道標記無法查證，行動者則需被阻止前往，以防浪費人力。
- **對「AI 腦補過度」的防範與不信任**：使用者不信任 AI 自動補齊的資訊（如自動補上時間、推測位置），希望系統能明確區分出「哪些是原文、哪些是 AI 推測」。

## 三個 persona 之間的衝突需求

- **回報速度（極簡） vs 資訊完整（極繁）**：
  - _回報者_ 希望越快越好，能在慌亂中送出片段消息、不完整文字或無時戳截圖；
  - _整理者與行動者_ 則需要詳盡的聯絡電話、精確坐標、時間與當事人公開授權，否則難以查核與出發。
- **公開透明（效率） vs 隱私安全（個資）**：
  - _行動者_ 希望所有災情與位置全數公開、定位於地圖上以方便導航；
  - _回報者（代通報者）_ 則擔心在未取得當事人同意前（如 M-011 長者案），直接在公用平台上暴露其精確住址與電話會帶來嚴重的個資外洩風險。

## Phase 0 原型中被推翻的假設

- **原本假設**：原以為將所有功能集中在一個「工作台」能簡化流程。
  - **被什麼回饋推翻（事實發現）**：被使用者共同的回饋推翻。單一介面反而造成角色混淆。回報者覺得工作台過於複雜，行動者誤讀「已整理」標籤為「已執行/可出發」，證明了介面必須根據角色進行分流（Role-based UI），而非所有人都看同一套 CRUD 介面。
- **原本假設**：只要把原始資料交給 AI 分析，就能自動幫整理者補齊欄位並生成高品質草稿。
  - **被什麼回饋推翻**：被整理者與行動者推翻。AI 對無時戳截圖（M-005）或流言（M-004）會自動「腦補」出合理的今天日期與高信心度，這反而會將錯誤資訊偽裝成高品質任務，誘騙志工前往。
- **原本假設**：使用綠色的「已整理」標籤表示該筆通報已被處理，是良好的進度狀態。
  - **被什麼回饋推翻**：被行動者推翻。行動者在清單看到綠色「已整理」會直覺認為「這任務已核實、可出發」，但點進去卡片才發現寫著「不可直接行動（待確認）」，這種標籤文案造成嚴重的安全認知落差。

## 最容易造成誤導的風險

- **狀態標籤語意混淆**：將「草稿已編輯（已整理）」與「現場狀況已核查（可行動）」混為一談，易誘導志工盲目出發。
- **儀表板以 sourceType 代表可信度**：儀表板將 `official_notice` 預設為高可信、`social_post` 為低可信，但事實上官方封路截圖可能早已過期（M-005），而社群通報可能非常即時且精確（M-010）。
- **混亂的 updatedAt 時間戳記**：未註明該時間是「貼文時間」、「進系統時間」還是「草稿編輯時間」，使用戶無法進行時效判定。

## 可以考慮的設計方向

- **生命週期狀態機（State Machine）**：徹底區分「整理進度」與「查證狀態」，將狀態細化為：`未處理` -> `整理中` -> `需人工複核` -> `已查實可行動` -> `已過期/結案`。
- **目擊與查證資訊解耦**：在回報端加入「目擊類型」（親眼目睹/聽人轉述）選單；在整理端加入「人工查證欄位」（查證人、電聯紀錄、核實時間）。
- **隱私防護開關**：加入「是否已取得當事人同意公開個資」的勾選，若未同意則在前端畫面自動模糊精確地址與電話。
- **AI 標註與人工審查確認**：所有經由 AI 生成的欄位在儲存前，必須在 UI 上用黃底標記「此欄位由 AI 推測，請人工核實」，且需要整理者手動確認才能儲存。

## 需要人類決策的取捨

- **取捨 1：回報表單的輸入門檻**
  - **選項 A**：極簡表單，只留一兩個輸入框，讓回報者快速送出，查證壓力留給後端整理者。
  - **選項 B**：嚴格表單，限制必須勾選目擊類型、留電話、輸入具體位置，否則無法送出。
  - **我目前傾向**：選項 A。災害當下收集線索速度最重要，若門檻太高使用者會放棄回報；但需在整理者畫面加強「未查證」之警示。
- **取捨 2：個資與地址的公開層級**
  - **選項 A**：預設完全模糊精確地址（如僅顯示到路段），只有核實且登入的志工才能查看完整位置。
  - **選項 B**：預設公開完整地址以便於地圖定位與運送。
  - **我目前傾向**：選項 A。安全與隱私大於便利，必須嚴防受災者個資外洩被不當利用。

## 不應該讓 AI 自動決定的事情

- **決定任務是否「可行動」**：是否派志工出發必須由人工查實決定，AI 絕對不能因為社群文字寫得完整就自動判定為「可出發」。
- **自動補齊時間與位置**：若原文無時間或無具體地點，AI 不得「腦補」日期或推測坐標，必須保持留空（null）或標記為「待確認」。

## 我採用的 AI 整理

- 採用了三個 Persona 關於「時效性」與「對 AI 腦補的擔憂」的彙整，這非常符合現場實際操作會碰到的痛點。

## 我不採用的 AI 建議

- **建議**：AI 在 Phase 0 曾建議根據來源 `sourceType` 為 `official_notice` 時自動將查核狀態設為 `verified`。
  - **不採用原因**：官方消息亦會過期（例如早上的封路通知，下午已解除），自動設為 verified 會誤導行動者。

## 我自己補充的判斷

- **角色介面分離**：必須在前端畫面上區分「回報端」、「整理端」與「行動端」三個視角或路由，避免將這三組互相衝突的需求塞在同一個大雜燴畫面中。

## 我仍不確定的地方

- 如果回報者只留下一句極度模糊的「聽說老街比較需要水電」（M-003），在完全沒有聯絡方式的情況下，資訊整理者除了將其標記為「無法查證/不處理」之外，是否有一套機制能讓其他在老街附近的志工順路去探查？這會不會帶來安全隱患？

## 小隊討論後的補充

- **sub-agent 訪談模擬非常有助於在沒有真實使用者的情況下，逼迫開發團隊站在「受災者」與「前線志工」的立場思考。它暴露了我們在 Phase 0 中對資料做出的「過度樂觀假設」。**
- **我們小隊一致認為，不能讓 AI 自動決定資料的「可行動性（Actionability）」，且應在 UI 上加強「AI 推測」與「人工確認」的視覺對比。**
